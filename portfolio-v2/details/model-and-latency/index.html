Model & Latency

模型不是越强越好：如何在能力与实时性之间取舍

Moqi 的实时任务是理解简短状态摘要、结合偏好选择表达，并生成可被程序调用的短回复。及时性本身就是产品能力。

任务与评估维度

模型选择不只比较综合能力，而是围绕桌面陪伴的实时链路做取舍。

Realtime Tasks

实时模型负责

  • 理解桌面状态摘要
  • 结合偏好判断表达方式
  • 生成短而自然的中文反馈
  • 输出结构化结果
Evaluation

重点评估

  • 响应延迟
  • 中文语境理解
  • 短回复质量
  • 输出约束稳定性
  • 调用成本

方案比较

实时交互与复杂任务不必由同一个模型完成。

方案优势问题当前用途
快速非思考模型延迟低、成本可控复杂推理能力有限实时策略与短回复
推理模型复杂判断能力强延迟较高后续记忆总结或复杂冲突
工具调用 Agent可完成搜索与外部任务延迟长、不可控步骤多当前实时链路暂不使用

版本迭代

从功能优先转向体验收敛，核心是让机器人在用户仍然期待反馈时作出回应。

V0 · 功能优先

部分交互 > 10 秒

接入搜索、时间、日历等工具，多次判断与网络请求破坏即时陪伴感。

V1 · 体验收敛

演示约 1.65 秒

删除非核心工具、限制搜索与回复长度，优先使用快速模型。

Current

核心链路接近 1 秒级

继续调整系统提示词、云端部署与服务配置。

当前决策

实时链路使用 DeepSeek 的快速非思考模型保障反馈；复杂记忆总结可在非实时链路异步处理。具体 API 型号待项目核实后填写，不在页面中猜测。
