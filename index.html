Project Overview

一个把桌面上下文、主动交互和机器人表达连起来的产品原型

这个项目不是普通提醒工具，也不是只会聊天的桌面助手。它关注的是：一个放在电脑旁边的机器人，如何理解用户当下的学习 / 工作节奏，并在合适的时候用合适的表达方式提供陪伴和支持。

项目目标 让电脑旁边的 AI 从被动问答变成懂节奏的桌面伙伴

面向长时间独自学习、写作、办公和远程协作的人群，设计一个能观察桌面节奏、主动陪伴、逐步学习偏好的机器人原型。

我的角色 产品定义、交互策略、前后端原型与硬件联调

独立梳理目标场景，设计主动交互规则、记忆机制和反馈分级，并把本地 Agent、云端后端和桌面机器人硬件串成原型链路。

核心产出 Agent 链路、主动策略、记忆机制和机器人反馈

已完成本地桌面状态采集、后端主动机会判断、用户反馈记录、记忆策略更新，以及表情 / 字幕 / 动作的硬件反馈链路说明。

适配方向 AI 产品、HRI 产品、智能硬件 / 具身智能产品

项目重点体现主动式 AI 的产品边界、桌面上下文理解、低压力陪伴、机器人化反馈和长期个性化学习。

Problem

电脑前的人不缺 AI 工具，缺一个懂节奏的桌面伙伴

研究生和办公人员经常长时间坐在电脑前：资料、论文、代码、会议和消息不断切换。真正影响体验的不是某一个提醒没响，而是低能量、任务漂移、孤立工作和 AI 工具过于被动带来的节奏断裂。

普通 AI 工具等用户主动打开、主动提问，无法感知当前桌面节奏，也很难在用户低能量或任务漂移时自然出现。
本项目方案把电脑上下文、机器人状态和用户反馈放进同一条链路，让机器人通过表情、字幕、轻动作或语音成为低压力桌面伙伴。

What I Built

最终产出是一套可以跑通的主动陪伴原型链路

页面展示的是当前已经完成的项目结构：本地桌面 Agent 负责观察，云端后端负责判断主动机会和反馈方式，记忆模块记录偏好与反馈，机器人硬件负责把策略变成可感知的表情、字幕和动作。

01

本地桌面 Agent

采集前台窗口、键鼠活跃、AFK、系统音频、ActivityWatch 等非摄像头信号，形成桌面上下文摘要。

02

主动陪伴策略

后端将桌面上下文转成主动机会，再根据状态、风险、偏好和机器人可用性选择反馈方式。

03

个性化记忆机制

记录任务上下文、互动反馈、场景偏好和节奏摘要，让机器人逐步学习什么时候适合出现。

04

机器人硬件联调

基于 ESP32 桌面机器人硬件完成后端接入，使策略结果能落到表情屏、底部字幕和头部动作。

Student Scenario

以研究生学习场景为主线：越用越懂我

学生场景更能体现这个项目的初衷：用户不是一直需要效率工具，而是在长时间独自学习时需要一个有存在感但不过度打扰的桌面伙伴。

Core Scene

长时间学习后的低能量状态

当用户长时间写论文、查资料或改代码，输入速度下降、窗口切换变多、停留但无明显推进时，机器人不做情绪诊断，只用轻动作、短字幕或任务拆分建议提供低压力陪伴。

System Logic

从桌面上下文到机器人反馈，再回到记忆更新

桌面上下文不是最终卖点，而是实现主动陪伴的路径。Agent 负责把电脑状态变成可解释的上下文，策略层负责控制表达边界，机器人端负责把结果变成具身反馈。

01桌面信号前台窗口、键鼠、AFK、系统音频、应用使用时长
02Agent 上下文学习、写作、会议、任务漂移、低能量、返回电脑
03主动机会是否需要陪伴、任务恢复、节奏支持或静默记录
04反馈策略根据状态风险、偏好、冷却和机器人可用性选择强度
05机器人表达表情、轻动作、底部字幕、语音播报
06记忆更新接受、忽略、关闭、延后、纠正和场景偏好

Interaction Strategy

低打扰不是项目主卖点，而是主动陪伴必须遵守的边界

机器人越主动，越需要克制。策略层不会让 Agent 直接决定高打扰行为，而是通过规则、置信度、用户偏好和硬件状态选择最低可行反馈强度。

quiet会议、深度专注、信号不足：只记录，不主动出现。
subtle低风险陪伴：表情、灯效或轻动作，弱存在感。
subtitle任务恢复、节奏建议：用短字幕表达，不直接语音打断。
speak空闲、高置信、重要事项：才允许语音播报。
Case A

会议中检测到主动机会

状态会议软件、麦克风或摄像头信号提高会议置信度。

策略普通语音被压制，必要信息降级为静默记录或短字幕。

价值机器人保持陪伴感，但不制造社交尴尬。

Case B

学习中出现任务漂移

状态窗口切换增多、输入下降、学习上下文被打断。

策略先记录上下文，适合时用短字幕帮助用户重新聚焦。

价值不是催促用户，而是降低重新进入任务的心理负担。

Memory Mechanism

记忆让机器人从“能主动”变成“越用越懂我”

系统不默认保存所有行为，而是围绕任务相关性、重复场景、用户反馈和时间衰减管理记忆。它学习的不是隐私细节，而是用户更愿意在哪些场景被怎样陪伴。

记录 -> 总结 -> 合并 -> 调用 -> 更新 -> 降权 把用户反馈和重复场景转化为可更新的策略权重，让主动陪伴逐步贴合个人习惯。
Context

短期上下文

当前任务、最近切换、学习片段和待处理主动机会。

Preference

长期偏好

偏好字幕、不喜欢语音、晚上减少工作提醒、喜欢简短表达。

Rhythm

行为节奏

高专注时段、容易低能量的时间、会议密集日和长时间学习模式。

Feedback

交互反馈

用户接受、忽略、关闭、延后或纠正后，更新场景策略和表达强度。

Hardware Integration

原型实现：机器人硬件联调

硬件部分不是概念图，而是基于 ESP32-S3N16R8-EMOJI 桌面机器人硬件与开源固件框架进行适配，把自有后端、WebSocket 指令和机器人表情 / 字幕 / 动作反馈串起来。

Prototype Chain

本地桌面 Agent -> 云端后端策略 -> 机器人反馈

当前原型已跑通桌面上下文上报、后端主动策略判断、机器人端表情和动作反馈的基本链路。

Firmware Adaptation

自有后端接入

OTA / WebSocket 指向自己的服务器，机器人不再依赖默认云端入口，便于验证主动陪伴策略。

Embodied Feedback

表情、字幕、头部动作

后端可下发 expression / motion / status，硬件端显示表情屏、底部字幕，并执行点头、摇头、转向等动作。

Product Boundary

基于开源框架适配

项目工作重点放在产品定义、系统整合、后端策略、本地 Agent、固件适配和联调，不把底层硬件框架包装成从零原创。

Reflection / Value

项目价值在于把主动 AI、桌面上下文和具身反馈做成一个可解释闭环

这个项目最重要的不是“多一个提醒功能”，而是探索一个桌面机器人如何在学习 / 办公场景中感知节奏、选择表达、学习偏好，并把陪伴感控制在合适强度。

AI 产品判断把 Agent 能力放进可解释的状态规则和用户偏好里，而不是让模型直接决定高打扰行为。
HRI 交互策略把机器人表达拆成表情、轻动作、字幕和语音，控制存在感与打扰感之间的平衡。
具身智能落地不只停留在聊天界面，而是让桌面状态最终影响硬件表情、姿态和字幕。
工程原型闭环本地 Agent、云端后端、记忆模块、硬件固件和作品集展示保持一致。
后续可补充材料 真实演示视频、硬件联调截图、更多学习场景样本、记忆编辑界面和长期使用反馈。
