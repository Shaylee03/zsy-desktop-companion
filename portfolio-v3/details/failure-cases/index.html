Failure Cases

从真实原型中发现问题，而不是只完成理想流程

每个 Failure Case 都被拆分为现象、风险判断与调整方案，用具体产品决策替代“更自然一点”这样的模糊反馈。

Case 01

单一反馈不明显

现象

只显示表情或字幕时，用户不容易注意到机器人正在主动表达。

判断

低打扰不等于越弱越好；无法被感知的反馈会失去价值。

调整

改为表情与短字幕组合，需要注意时加入内置动作，重要场景才用语音。

Case 02

工具调用延迟过长

现象

搜索与日历任务经历多次判断和网络请求，部分交互超过十秒。

判断

功能范围已经偏离当前阶段的核心产品目标。

调整与结果

删除非核心工具、限制搜索、缩短回复并优化部署，核心链路降至约一秒级。

Case 03

不确定状态下错误打扰

现象

短暂停止输入也可能是在思考或阅读，不一定表示空闲。

风险

把单一信号直接解释为空闲，会导致机器人在错误时机出现。

调整方向

综合应用、持续时间与音视频状态；低置信度静默，在自然节点重判。仍需测试集验证。

Case 04

一次行为被过度解释

现象

一次关闭语音可能只是临时情况，并不代表长期偏好。

风险

过早形成记忆会错误改变后续策略。

调整

单次隐式行为只形成低置信候选；明确确认和重复行为才提升置信度。

Case 05

用户无法纠正理解

现象

仅用隐式反馈更新时，用户不知道机器人形成了什么判断。

风险

错误偏好可能持续影响之后的交互。

调整方向

增加记忆管理页面，允许查看、修改和删除；完整流程仍在完善。
