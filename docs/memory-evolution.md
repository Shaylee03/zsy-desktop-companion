# 记忆进化机制

## 设计目标

本项目里的记忆用于让机器人持续修正主动介入用户学习和工作的边界。

记忆必须满足 4 个条件：

1. 可解释：能说明某次主动介入为什么被降级、延后或取消。
2. 可查看：用户和调试后台能看到关键偏好和策略状态。
3. 可更新：用户行为变化时，旧记忆不能永久支配策略。
4. 可降权：长期不用、冲突、低置信或被用户纠正的记忆要降权或遗忘。

## 四类记忆

| 类型 | 记录内容 | 主要用途 | 不应该记录 |
| --- | --- | --- | --- |
| 短期上下文记忆 | 离席前任务、最近窗口切换、当前待处理主动机会、最近工作片段 | 离席返回摘要、任务恢复、上下文连续 | 私密消息正文、完整窗口标题流水 |
| 长期偏好记忆 | 不喜欢语音、会议禁声、晚上不做工作介入、偏好字幕、喜欢简短表达 | 控制反馈方式、语气和默认打扰强度 | 一次性情绪、低置信猜测 |
| 行为节奏记忆 | 常见专注时段、会议密集日、休息接受时机、疲劳模式 | 优化介入时机和每日预算 | 对用户生产力做道德化评价 |
| 交互反馈记忆 | 接受、忽略、关闭、纠正、延后、是否最终有帮助 | 调整主动介入边界和场景权重 | 未经解释的模型黑箱权重 |

## 生命周期

```text
记录 -> 总结 -> 合并 -> 调用 -> 更新 -> 降权 / 遗忘
```

### 1. 记录

系统先记录原始事件，不直接把一次行为变成长期结论。

示例：

```text
用户在会议中连续关闭语音反馈。
用户在写代码时多次忽略休息语音，但接受字幕提示。
```

### 2. 总结

当同类事件重复出现后，系统把事件压缩成偏好或策略建议。

示例：

```text
用户在会议场景中偏好非语音反馈。
用户写代码时更适合 subtitle 或 subtle，语音反馈保持更高触发门槛。
```

### 3. 合并

相似记忆要合并，避免同一偏好散落成多条互相冲突的记录。

示例：

```text
“Zoom 中关闭过语音”
“Teams 中忽略过语音”
“会议时说过别出声”

合并为：用户在会议场景中默认禁止 speak，可允许 quiet 或 subtitle。
```

### 4. 调用

相似场景再次出现时，策略层调用相关记忆，但不能绕过硬边界。

示例：

```text
检测到会议中 -> 调用会议偏好 -> 普通主动机会从 subtitle/speak 降级为 quiet。
```

### 5. 更新

当用户行为改变时，记忆内容和置信度要更新。

示例：

```text
用户后来主动开启会议字幕反馈。
策略更新为：会议中允许 subtitle，但继续禁止 speak。
```

### 6. 降权 / 遗忘

长期不用、过期、冲突、低置信或被用户纠正的记忆不应继续高权重影响策略。

示例：

```text
用户过去不喜欢休息介入，但最近连续接受休息建议。
旧偏好降权，新偏好提高置信度。
```

## 生成式能力与规则边界

生成式能力可以负责：

- 总结离席期间发生了什么。
- 把行为日志压缩为记忆候选。
- 检查冲突记忆。
- 生成阶段性行为总结。
- 帮用户拆解任务。

生成式能力不直接负责：

- 是否突然语音播报。
- 是否在会议中主动介入。
- 是否把一次行为写入长期记忆。
- 是否根据低置信状态做强判断。

项目原则是：

> 生成式能力负责理解与表达，产品规则负责边界与约束。

- 更稳定的任务恢复摘要。
- 明确的记忆冲突检测和降权策略。
- 记忆命中率、用户纠正率、冲突记忆率的标注口径。
- 面向用户的记忆查看、修改和删除流程。
